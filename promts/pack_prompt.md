# ПРОМТ ДЛЯ УПАКОВКИ ПРИЛОЖЕНИЯ (ВЕРСИЯ 2.0)

## ЦЕЛЬ:
Упаковать приложение в файл source.py для автоматического развертывания через deploy_app.py

## ВАЖНЫЕ ОСОБЕННОСТИ СИСТЕМЫ:
- deploy_app.py автоматически находит и обрабатывает ВСЕ файлы source*.py
- Файлы обрабатываются в порядке: source.py, source1.py, source2.py...
- При конфликтах файлы из более поздних sourceX.py перезаписывают предыдущие
- pack_app.py автоматически исключает служебные файлы (source*.py, pack_app.py, deploy_app.py)
- НОВЫЕ ФАЙЛЫ можно добавлять в КОНЕЦ существующего source.py (они перезапишут старые версии)

## ФОРМАТ УПАКОВКИ:
Каждый файл оборачивается в маркеры:
# ======== FILE: путь/к/файлу ========
(ПОЛНОЕ содержимое файла — от первого до последнего символа)
# ======== END FILE: путь/к/файлу ========

## КРИТИЧЕСКИЕ ТРЕБОВАНИЯ:

### 1. ТОЛЬКО ПОЛНЫЕ ФАЙЛЫ
- ЗАПРЕЩЕНО использовать "..." или "[код не изменился]"
- ЗАПРЕЩЕНО пропускать части файла
- ЗАПРЕЩЕНО отправлять только различия (diff)
- КАЖДЫЙ файл должен быть ПОЛНОСТЬЮ от первого до последнего символа

### 2. РАБОТА С README.md
- README.md должен быть ПОЛНОСТЬЮ закомментирован
- КАЖДАЯ строка README.md начинается с символа "#"
- Пустые строки тоже начинаются с "#"
- Пример правильного формата:
# ======== FILE: README.md ========
# # Заголовок
#
# Текст описания
# ======== END FILE: README.md ========

### 3. ПРАВИЛА УПАКОВКИ
- Сохранять исходную иерархию папок
- Кодировка: UTF-8
- Отступы: 4 пробела (без табов)
- Имя файла: source.py (не my_source.py)

### 4. ЧТО ДЕЛАТЬ ПРИ ОБНОВЛЕНИИ
- Если нужно изменить несколько файлов → упаковать ТОЛЬКО изменённые файлы
- Новые версии можно добавить в КОНЕЦ существующего source.py
- Или создать source1.py, source2.py... для независимых версий
